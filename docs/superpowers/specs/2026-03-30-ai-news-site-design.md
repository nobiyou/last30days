# AI 资讯站设计

日期：2026-03-30

## 概述

建设一个垂直聚焦的 AI 资讯站，服务对象是独立开发者、创业团队和投资人。

第一版是一个全自动、滚动更新的资讯站，覆盖以下主题：

- AI
- 机器人
- 智能体
- 开源

站点会自动发布短快讯卡片，按滚动方式持续更新，并将每张卡片沉淀到对应的 Topic Hub 页面中。`last30days-skill` 作为研究与信号引擎复用，站点侧的选题发现、编辑归一化和发布层单独建设。

## 产品目标

这个产品的首要价值是速度和实用性。

用户打开站点后，应该能很快回答以下问题：

- 刚刚发生了什么值得关注的事？
- 为什么这件事对开发者或投资人重要？
- 这条信息属于哪个主题簇？
- 如果我想继续深入，应该从哪里看起？

这个产品有意优先做成“信息产品”，而不是一个通用 CMS、仪表盘或媒体品牌系统。

## 目标用户

主要读者：

- 独立开发者
- 创业团队
- 投资人

这些用户的共同需求是：

他们需要快速发现信号和高价值上下文，而不是泛泛的科技新闻。这个站点应当更像 AI 行业信号流，而不是传统的综合媒体站。

## 产品形态

站点采用 `滚动快讯流 + Topic Hub` 的模式。

### 核心页面

1. 首页 / Latest

- 按时间倒序展示滚动信息流
- 强调最新、最强的信号
- 适合高频回访

2. Topic Hub 页面

- 示例主题：`Agents`、`Robotics`、`Open Source`、`AI Infra`
- 每条发布的卡片都会自动归入一个或多个 Topic Hub
- Topic Hub 会随时间不断累积内容，并为后续更深度的内容扩展提供基础

3. 快讯详情页

- 每张快讯卡片都有一个永久详情页
- 存放结构化元数据和来源上下文
- 作为该事件的 canonical permalink

## 发布单元

系统的核心内容单元是“短快讯卡片”。

### 卡片格式

每条发布卡片应控制在约 100-200 字，并遵循稳定的结构：

- 发生了什么
- 为什么重要
- 谁应该关注
- 来源信号

这种结构可以保证输出稳定、更适合自动化，也能避免站点退化成单纯的链接堆砌。

## 为什么适合复用 `last30days`

`last30days-skill` 很适合作为底层引擎，因为它已经提供了：

- 多源采集
- 相关性与时效性评分
- 去重
- 基于 SQLite 的持久化存储
- 用于持续跟踪的 watchlist 和 briefing 思路

它的强项是信号发现和研究归纳。

相对这个项目而言，它的弱项在最终发布层。它并没有提供一个可直接用于生产的网站发布层、编辑状态模型，或适合内容站自动发布的安全格式化能力。

因此推荐架构是：

- 复用 `last30days` 做研究
- 站点侧单独建设发现控制、卡片生成、路由和发布流水线

## 架构

整个系统分为四层。

### 1. 发现层（Discovery Layer）

目的：

- 找到“现在值得研究”的候选主题

输入：

- 精选领域 watchlists
- 与 AI、机器人、智能体、开源相关的趋势来源
- 主题 / 实体聚类

输出：

- 排序后的候选事件或主题，交给研究层处理

### 2. 研究引擎层（Research Engine）

目的：

- 跨来源收集并归一化原始信号

实现方式：

- 复用 `last30days.py`
- 在可行范围内复用已有的数据源适配器、评分、去重和存储逻辑

输出：

- 包含分数、时间戳和来源引用的结构化 findings

### 3. 编辑归一化层（Editorial Normalizer）

目的：

- 将 findings 转换成可以直接发布的短快讯卡片

职责：

- 判断某条 finding 是否值得发布
- 生成标题
- 生成短摘要
- 解释为什么重要
- 打上正确的 Topic Hub 标签
- 生成稳定的事件标识

这一层是 MVP 的关键新增层。它是“原始研究结果”转化为“站点内容”的核心环节。

### 4. 发布层（Publisher）

目的：

- 生成站点产物并完成部署

职责：

- 更新首页信息流
- 生成详情页
- 更新 Topic Hub 页面
- 输出静态内容
- 部署最新构建结果

## 系统边界

### 直接复用或轻量改造

- `last30days.py`
- 各数据源适配器
- 评分逻辑
- 去重逻辑
- SQLite 持久化思路
- watchlist 和 briefing 模式

### 新建部分

- 面向站点的主题发现
- 发布门槛逻辑
- 快讯卡片生成器
- Topic 路由
- 静态站点生成与部署

### 明确延后

- 人工编辑审核
- 社交分发
- 长文分析工作流
- 用户账户
- CMS 后台

## 发现策略

第一版不应尝试无限制的全网自由发现。那样会过早引入过多噪音。

更合适的方式是构建一个可控的候选池，来源包括：

- 精选领域 watchlists
- 目标赛道中的已知项目和实体
- GitHub、Hacker News、Reddit、X 以及精选技术信息源等趋势面

发现层应先对这些输入做聚类，形成候选事件或主题，然后再调用 `last30days`。

这样既能让覆盖面保持足够新鲜，又能兼顾质量。

## 滚动更新模型

站点应当采用滚动更新，但不追求真正实时。

推荐节奏：

- 每 15-30 分钟运行一次发现与发布任务

这样既能保持“正在更新”的产品感，又不会引入真正流式处理带来的复杂度和脆弱性。

每次循环做两件事：

1. 发现候选主题或事件
2. 对合格候选做研究、归一化并发布卡片

## 发布门槛

不是每条 finding 都应该成为卡片。

一条内容只有在满足以下事件条件之一时，才允许发布：

- 多源共振：同一事件在两个及以上有意义的来源中同时出现
- 单源强爆发：某个高可信来源表现出异常强的互动量或重要性
- 结构性更新：版本发布、融资、开源发布、重要 issue、重要论文、产品发布或类似变化事件

并且同时满足以下基础条件：

- 事件足够新，值得当下关注
- 事件明确属于站点覆盖领域
- 事件不是已发布卡片的重复项
- 置信度分数达到最低阈值

整个系统应当偏保守。

原则：

- 宁可错过一条弱卡片，也不要自动发布低置信度内容

## 内容对象模型

每个通过发布门槛的事件，在渲染为页面之前，都应先归一化成一个稳定的内容对象。

最小字段：

- `title`
- `summary`
- `why_it_matters`
- `who_should_care`
- `topic_tags`
- `source_links`
- `confidence_score`
- `published_at`
- `canonical_event_id`

这个归一化对象会成为以下输出的单一事实来源：

- 首页卡片
- 详情页
- Topic Hub 归档
- 后续扩展，如 RSS、邮件或社交分发

## 发布产物

每次成功发布时，需要同时更新三个界面：

1. 首页 Latest feed
2. 该卡片的永久详情页
3. 一个或多个 Topic Hub 页面

因此，发布并不是“往首页追加一条内容”这么简单，而是基于一个 canonical 内容对象，同时写入多个内容面。

## 失败处理

整个流水线必须支持安全降级。

### 来源失败

- 某一个来源失败时，继续使用其他来源
- 不允许因为单个上游来源不可用，就让整轮发布失败

### 研究结果过弱

- 如果 findings 太弱或噪音过大，则记录到候选层，但不发布

### 生成失败

- 如果编辑归一化层无法生成完整的高质量卡片，则退回受约束的模板化摘要
- 绝不发布空白或格式损坏的内容

### 部署失败

- 保持上一版站点继续在线
- 下一轮调度时自动重试

### 重复事件

- 如有必要，更新已有事件的聚合信号
- 除非产品后续明确支持“跟进更新”，否则不要为同一事件再次发布第二张卡片

## MVP 技术形态

第一版应尽量保持简单和运行稳定。

推荐的 MVP 技术组合：

- Python 负责编排和研究集成
- SQLite 作为短期结构化内容源，复用现有模式
- 静态站点生成用于发布
- Cron 或等价调度机制负责周期执行

MVP 应优先追求确定性输出和低成本部署，而不是高度灵活性。

## MVP 范围

### 范围内

- 滚动首页 feed
- 快讯详情页
- Topic Hub 页面
- 自动发现候选主题
- 自动研究
- 自动生成卡片
- 自动静态发布
- 去重
- 发布门槛控制
- 基础重试与安全兜底

### 范围外

- 人工审核后台
- 多用户工作流
- 评论
- 认证系统
- 实时 socket 推送
- 社交自动分发
- 长文生成
- 完整 CMS 体验

## 验收标准

如果 MVP 能证明以下结果，就可以视为成功：

- 它能稳定自动发布至少一周
- 它能持续产出有用且不空洞的滚动更新
- 重复输出能保持在可控范围内
- 明显低置信度内容通常会被正确过滤
- Topic Hub 会随时间积累出有意义的内容
- 对目标读者而言，站点看起来有价值，而不是普通聚合站

## 验证标准

在宣称 MVP 可用之前，应通过以下运行检查：

1. 调度稳定性

- 发布循环可以反复运行，不需要人工持续看护

2. 发布质量

- 卡片持续满足 100-200 字短快讯格式要求

3. 安全性

- 生成失败或部署失败不会导致线上站点损坏

4. 内容完整性

- 重复事件不会造成失控的重复发布

5. 产品实用性

- 对目标读者来说，输出值得每天打开一次

## MVP 之后的路线图

当前选择的站点形态，刻意保留了清晰的升级路径。

未来可扩展方向包括：

- 更深的 Topic 页面
- 工具对比
- 开源项目拆解
- 实战 workflow
- 精选 Newsletter
- 社交分发

因为 Topic Hub 本身就在 MVP 中，所以未来增加更深的编辑内容时，可以直接沿用同一套信息架构，而不需要重做站点结构。

## 决策摘要

当前确认的方向是：

- 垂直 AI 资讯站
- 受众：独立开发者、创业团队、投资人
- 主格式：短滚动快讯卡片
- 站点形态：滚动 feed + Topic Hub
- 发布模式：全自动
- 更新模式：滚动更新，而非实时流
- `last30days` 的角色：研究与信号引擎，而不是最终发布系统

这个设计让第一版保持聚焦、实用且技术上可达成，同时也为后续演进成更丰富的内容产品保留了清晰空间。
